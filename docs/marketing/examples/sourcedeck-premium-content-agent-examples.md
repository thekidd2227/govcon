# SourceDeck Premium Content Agent — Worked Examples

> **Illustrative only.** These examples show the *structure* of a draft the agent should produce. The business details, features, and claims below are placeholders chosen to demonstrate the format. They are not real SourceDeck customer content and not approved for publishing.

Each example follows this structure:

- **Input source** — what the user attached or connected
- **Extracted insight** — what the agent pulled from the evidence
- **Recommended post type** — one of the supported content type identifiers
- **Draft post** — the actual draft body
- **CTA** — call to action and destination
- **Hashtags** — per the platform hashtag schema (10–12 LinkedIn, 3–6 Facebook)
- **Claim confidence** — `verified`, `inferred`, or `candidate` per claim

## 1. LinkedIn text-only authority post from a GitHub README

- **Input source:** GitHub repository link, README at root
- **Extracted insight:** The user's product has a documented "document control" module that tracks proposal versions, approvals, and submission deadlines. README explicitly describes the feature and use case.
- **Recommended post type:** `text_authority`

**Draft post:**

> Most small contractors lose proposals at handoff, not at writing.
>
> The version that goes out at 11:58 PM is often not the version the capture lead approved at 4 PM. Different file. Different paragraph. Same template name.
>
> Document control is the part nobody calls a "feature." It is the part that decides whether your proposal actually reflects what your team agreed to submit. Tracked versions. Named approvers. Submission deadline tied to the document, not the email thread.
>
> If your team cannot answer "which version went out?" in under 30 seconds, you do not have a writing problem. You have a control problem.

**CTA:** Visit `sourcedeck.app/document-control` to see how the module works.
